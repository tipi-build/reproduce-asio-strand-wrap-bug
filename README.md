# Strand wrapped completion tokens do not result on dispatch on the strand anymore since the removal of asio_handler_invoke hooks

The documentation of Boost.Asio invites users of the deprecated asio_handler_invoke ADLs hook to change in favor of the associated_executor traits.

Since Boost 1.84.0 asio_handler_invoke ADL hooks were completely removed. We noticed transitioning a codebase from Boost.Asio 1.81.0 to Boost.Asio 1.85.0 that the abstraction does not easily get replaced.

Indeed as you will see in the attached code, the code had a proxy base class type which was used in C++03 code to centralize code for handler proxying. This is a highly simplified + anonymized version of production code. Intermediate completion handlers were built thanks to this idiom and the ADL hooks with base class relationship would trigger to handle proxying the call to the underlying handlers.

It happens that this Base class relation ship really doesn’t play well when switching from asio_handler_invoke to associated_executor. It is an independent discussion, but if [associated_executor](https://www.boost.org/doc/libs/1_83_0/doc/html/boost_asio/reference/associated_executor.html) was given an SFINAE selection parameter ( typename = void ) it would allow to migrate such code in an easier fashion. Right now AFAICT it’s not possible to associate an executor based on the handler base class without using private implementation detail:: of asio. 

But that’s not the core of the issue, after converting all the code to the asio without asio_handler_invoke data races started to show and assertions checking on `io_context::strand.running_in_this_thread` to trigger, while it did before yield true.

It happens that in the past, asio_handler_invoke_helpers::invoke was called when handler_work did not detect any owning executor or IoExecutor ( see https://github.com/chriskohlhoff/asio/blob/7609450f71434bdc9fbd9491a9505b423c2a8496/asio/include/asio/detail/handler_work.hpp#L518-L528 ).

This was the case when the handler to be called was a strand-wrapped ( Obtained by `strand.wrap()` call ) handler was called and the asio_handler_invoke_helpers::invoke code would use the strand-wrapped handler dispatcher to dispatch the execution ( c.f. https://github.com/chriskohlhoff/asio/blob/7609450f71434bdc9fbd9491a9505b423c2a8496/asio/include/asio/detail/wrapped_handler.hpp#L242-L244 ), which in the case of a strand wrapped handler would be the strand.

This now instead just a plain call to the function and doesn’t go through the strand anymore. strand::wrap is indeed deprecated, but the behaviour is very surprising and is only visible at compile time.
If one changes `strand.wrap` call with `bind_executor(strand)` the issue gets solved and `running_in_this_thread()` returns true again. 

I can’t seem to find any documentation on it, and possibly deleting `strand.wrap()`  will solve this issue, as it will force anyone to migrate to `bind_executor, but as of now with `strand.wrap()` still present, the behaviour is surprising.

but I wanted to have your thoughts on this, as it seems that where before strand guarantee would be provided transitively by checking the completion_token dispatcher, this doesn’t happen anymore.
